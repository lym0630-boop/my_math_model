# LPPO 总览与面试指南

## 一句话概括

> 在标准 GRPO 基础上，用 EMA 跟踪每道题的学习进度，动态调节 advantage 权重，让 GPU 算力集中在模型"正在学会"的题目上。

## 30秒面试话术

> "我在标准 GRPO 的基础上加了 Learning Progress 加权机制。核心观察是：GRPO 对所有题目等权训练，但实际上已经掌握的题（pass rate ≈ 1）和完全不会的题（pass rate ≈ 0）都贡献不了有效梯度，只有正在学习的题（pass rate 有正向变化）才是有效训练信号。
>
> 实现上，我用 EMA 跟踪每道题的 pass rate 变化趋势，计算一个 learning progress score，然后通过 sigmoid 映射到 [0.25, 2.0] 的权重区间，在 compute_advantage 之后乘以这个权重。
>
> 代码侵入性很小：训练循环里只加了两个 hook 点——一个更新状态和加权，一个保存 checkpoint。用 `++lppo.enable=False` 一行配置就能完全关闭。"

## 面试常见追问 & 回答

### Q1: 为什么用 EMA 而不是直接用当次 pass rate？
**A**: 直接用当次有两个问题：(1) 采样方差大——8 次 rollout 的 pass rate 波动很大，可能一次 5/8、下一次 1/8，不代表真实能力变化；(2) 没有"方向"信息——EMA 的差值（lp = p_new - p_old）天然提供了学习方向，正值表示在进步。

### Q2: 为什么 weight 乘 advantage 而不是乘 reward？
**A**: GRPO 的 advantage 已经做了组内标准化（减均值除标准差），如果我在 reward 层面就加权，会破坏 GRPO 的正确性——同一组内不同 rollout 的 reward 基线应该是公平对比的。在 advantage 层面加权，等价于说"这道题的梯度方向对，但我希望这个方向的步长大一点或小一点"。

### Q3: 这个 w_min = 0.25 是怎么选的？
**A**: 经验值。设为 0 会完全忽略某些题导致遗忘；设太大（比如 0.5）则区分度不够。0.25 意味着最不重要的题还保留 1/4 的训练信号，既省算力又防遗忘。上限 2.0 是因为梯度放大太多（如 5x）会导致训练不稳定。

### Q4: 和 Curriculum Learning 有什么区别？
**A**: Curriculum Learning 通常是按固定 schedule 调整数据分布（easy → hard），是"一刀切"的策略。LPPO 是在线自适应的——每道题的权重随着训练动态变化，同一道题可能从"学习中"变成"已掌握"然后权重下降。更精细且不需要人工设计 curriculum。

### Q5: 计算开销大吗？
**A**: 几乎可忽略。LP state 是一个 Python dict，120K 道题也只占几十 MB 内存。每个 step 只做简单的 float 运算（EMA + sigmoid），相比 GPU 上的 forward/backward 完全不在同一量级。

### Q6: PG (Prefix-Guided) 样本是怎么回事？
**A**: 对于完全不会的题（hard-zero），模型连开始推理的方向都找不到。PG 的思路是：把参考解的前 40% 作为前缀放在 assistant 消息中，让模型从中间开始续写。如果续写正确，就获得了正向 reward 信号，模型就知道"这个推理方向是对的"。类似于 warmstart 中的前缀引导思想。

## 开关方式

```bash
# 开启 LPPO
++lppo.enable=True

# 关闭 LPPO（不影响任何其他逻辑）
++lppo.enable=False
```

---

## 关键超参数

| 参数 | 默认值 | 含义 | 调参建议 |
|------|--------|------|----------|
| `ema_beta` | 0.8 | EMA 衰减系数 | 越大越平滑，[0.7, 0.9] |
| `w_min` | 0.25 | 最小权重 | 不要设 0，[0.1, 0.4] |
| `w_max` | 2.0 | 最大权重 | 不要太大，[1.5, 3.0] |
| `sigmoid_k` | 10 | sigmoid 陡峭度 | 越大越像二值化 |
| `tau` | 0.08 | sigmoid 阈值 | score > tau 才开始加权 |
